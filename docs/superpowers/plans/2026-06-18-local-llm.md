> task-by-task executable plan. Resolve the prerequisites in §2 and confirm the
> model choice, then convert into a phased executable plan (see §7). No manifests
> are written until then.
>
> **Revised 2026-06-18:** model downgraded from gemma-4-12B-it to
> **gemma-4-E4B-it (8B)** with **on-demand loading**, because the box stays at
> 32 GB RAM and now also runs Immich (see the foundation plan
> `2026-06-18-proxmox-single-vm.md` §5). The 12B model no longer fits the RAM
> budget. The VM is the single 28 GB node from the foundation plan (not a
> separate 16 GB master).

**Goal:** Host a local LLM on the `pk3s` master VM, accelerated by the Intel Arc
B580 (Battlemage) via llama.cpp's SYCL backend, served through an
OpenAI-compatible API for a coding assistant. Internal-only access.

**Architecture:** A dedicated `llm` namespace on the master VM (which owns the
PCI-passthrough B580). A llama.cpp SYCL container runs `llama-server`, mounting
`/dev/dri`, loading a GGUF model from the Proxmox ZFS NFS share, and exposing an
OpenAI-compatible `/v1` endpoint (chat + FIM completions) behind
`llm.local`. The B580 is **shared** with Jellyfin (arr-stack plan) — model size
is deliberately kept ≤8 GB resident so transcodes don't OOM.

**Tech Stack:** raw Kubernetes manifests (matches the `floci` raw pattern),
llama.cpp (SYCL build for Intel GPU), `gemma-4-E4B-it` (8B) GGUF (primary),
Traefik IngressRoute (CRD), no SealedSecret needed (model weights are public).

---

## 1. Scope & decisions

| Aspect | Decision |
|--------|----------|
| Serving stack | **llama.cpp** (`llama-server`, SYCL build) — user preference |
| Backend | Intel Arc B580 via SYCL (not Ollama/IPEX-LLM/vLLM) |
| Model (primary) | **gemma-4-E4B-it** (8B, ~5 GB load) — fits the 32 GB box alongside Immich |
| Model (fallback) | Qwen2.5-Coder-7B (if gemma-4 arch unsupported in the SYCL build) |
| Loading | **On-demand** — pod scales down / model unloads when not coding, freeing RAM for Immich |
| Use case | Coding assistant (autocomplete via FIM + chat) |
| Hosting | k3s on the single VM (foundation plan), GitOps-managed |
| GPU | PCI-passthrough B580 into the VM, shared with Jellyfin; Immich ML stays on CPU |
| Exposure | Internal-only — `llm.local`, `web` entrypoint, no Cloudflare |

**Rejected stacks (why):**
- **Ollama** — works, but the user prefers llama.cpp directly (Ollama uses
  llama.cpp under the hood anyway).
- **IPEX-LLM** — **archived by Intel Jan 2026** (no maintenance/security fixes).
  Do not start new deployments on it.
- **vLLM (XPU)** — active, but no prebuilt XPU wheels (build from source,
  Python 3.12, triton-xpu). Overkill for single-user serving.

**Cluster reality:** the cluster is collapsing to a single Proxmox VM
(12 vCPU / 28 GB) with the B580 passed through — delivered by the foundation plan
(`2026-06-18-proxmox-single-vm.md`). This plan runs on top of that VM.

## 3. Model selection — why gemma-4-E4B-it (8B)

The B580 has **12 GB GDDR6 @ ~456 GB/s**. For batch-of-1 decode, throughput ≈
bandwidth ÷ model size, so smaller-and-fitting beats larger-and-offloaded. But
the **binding constraint is now host RAM, not VRAM**: the box stays at 32 GB and
also runs Immich (see foundation plan §5). The 12B model (~8 GB load) no longer
fits the RAM budget alongside Immich + monitoring + arr + Forgejo.

| Model | Q4 size | Fits VRAM | Fits 32 GB box + Immich? | Est. tok/s |
|-------|---------|-----------|--------------------------|------------|
| **gemma-4-E4B-it** (8B, eff. 4B) | ~5 GB | ✅ easily | ✅ **(primary)** | ~50–60 |
| gemma-4-12B-it (QAT) | ~7.5 GB | ✅ | ⚠️ only with strict on-demand + no Immich scan; **dropped** | ~35–50 |
| gemma-4-E2B-it (5B, eff. 2B) | ~3 GB | ✅ trivially | ✅ (weaker) | ~70+ |
| Qwen2.5-Coder-7B | ~5 GB | ✅ | ✅ (fallback) | ~50–60 |
| Qwen3-8B dense (general+thinking) | ~5 GB | ✅ | ✅ | ~50–60 |
| Qwen3-Coder-30B-A3B (MoE) | ~17 GB | ❌ partial | ❌ | ~10–25 |

**Primary: gemma-4-E4B-it (8B)** — fits the VRAM *and* the 32 GB host RAM budget
with margin, fast (~50–60 tok/s), leaves VRAM for Jellyfin, and frees RAM for
Immich. Combined with **on-demand loading** (unload when not coding), its RAM
footprint is ~0 most of the day.

**Why not the 12B anymore:** the ~8 GB model-load peak + Immich resident +
monitoring + arr pushes the VM past its 28 GB during simultaneous jobs. The 8B is
the robust choice given the foundation plan's 32 GB constraint.

**Fallback (if P4 fails — gemma-4 arch not yet in your SYCL build):**
- Qwen2.5-Coder-7B — coding-specialized, well-supported on Intel GPU, same ~5 GB.

## 4. Architecture

```
Proxmox host
├── B580 ──PCI passthrough──▶ master VM   (also serves Jellyfin in the arr plan)
│
└── ZFS tank/llm-models ──NFS──▶ master VM  (GGUF weights)
        │
   Namespace: llm   (nodeSelector: k3s-master)
   └── Pod: llama-server
         ├── container: llama.cpp (SYCL build), mounts /dev/dri, supplementalGroups:[render]
         ├── PVC: models  (NFS → Proxmox ZFS tank/llm-models)
         ├── env: GGML_SYCL_DEVICE=0, ...
         ├── Service: llama-server:8080  (OpenAI-compatible /v1)
         └── IngressRoute: llm.local  (internal-only, web entrypoint)
```

**Data flow:**
- Editor/Continue plugin (LAN) → `http://llm.local/v1/...` → Traefik → llama-server pod
- Cluster-internal callers → `http://llama-server.llm.svc:8080/v1/...`
- Model weights: pulled once to the NFS PVC, then memory-mapped by llama-server

## 5. llama.cpp serving detail

- **Container:** a llama.cpp **SYCL build** for Intel GPU. Pin a recent build that
  supports the gemma-4 architecture (P4). Community/official SYCL Dockerfiles
  exist; or build from source with `-DGGML_SYCL=ON` against Intel OneAPI/DPC++.
- **Runtime needs (in the container):** Intel GPU compute runtime recent enough
  for Battlemage (Xe2), matching the `xe` host driver. Mount `/dev/dri` from the
  passed-through B580; `GGML_SYCL_DEVICE=0`.
- **Binary:** `llama-server` → OpenAI-compatible endpoints:
  - `/v1/chat/completions` — chat / inline code Q&A
  - `/v1/completions` and `/infill` — **FIM autocomplete** for editor plugins
  - `/v1/embeddings` — if you later want local embeddings
- **Launch args (sketch):**
  ```
  llama-server \
    -m /models/gemma-4-12b-it-qat-q4_0.gguf \
    --host 0.0.0.0 --port 8080 \
    -ngl 99              # offload all layers to GPU (model fits fully)
    -c 8192              # context window
    -t 4                 # threads (GPU-bound; 4 is plenty for tokenization)
    --jinja              # use the model's chat template
  ```
  (For a fallback model that doesn't fit fully, tune `-ngl` down for partial offload.)
- **No partial offload needed** for the primary model — gemma-4-E4B-it fits fully,
  so `-ngl 99` puts everything on the B580 for max speed.

## 6. Resource budget (single VM, per foundation plan)

| | Primary (gemma-4-E4B 8B) | Notes |
|---|---|---|
| VM RAM | **28 GB** (foundation plan) | shared pool with all other workloads |
| vCPU | 12 (foundation plan) | tokenization + API serving only; GPU does decode |
| VRAM | ~5 GB resident (when loaded) | ~6 GB left for Jellyfin transcodes |
| Host RAM when loaded | ~5 GB (load peak) | **0 when idle (on-demand)** → frees RAM for Immich |

CPU-light. The binding constraint is **host RAM at 32 GB** (shared with Immich),
hence the 8B model + on-demand loading. See the foundation plan §5 for the full
box budget.

## 7. Build phases (gated)

| Phase | Contents | Gate to proceed |
|-------|----------|-----------------|
| **0** | Foundation plan delivers: single VM (12 vCPU / 28 GB) with B580 passed through | `/dev/dri/renderD128` visible; `xe` driver bound |
| **1** | `llm` namespace + models PVC (NFS → ZFS `tank/llm-models`) | a test pod can mount the PVC |
| **2** | llama-server pod (SYCL build), load a known-good GGUF first (Qwen2.5-Coder-7B) to prove the GPU path | `curl /v1/models` responds; tokens generate on the B580 (`intel_gpu_top` shows activity) |
| **3** | Swap to **gemma-4-E4B-it** GGUF (P4 gate) | loads cleanly; ~50–60 tok/s confirmed; FIM endpoint works |
| **4** | Wire up **on-demand loading** (scale-to-zero / cron / manual stop when not coding) + Service + `llm.local` IngressRoute | autocomplete + chat work from the LAN; pod RAM frees when idle |

Phase 2 de-risks the GPU/SYCL plumbing with a known-good model before depending
on the fresh gemma-4 architecture. Phase 4's on-demand loading is what makes the
8B model coexist comfortably with Immich on 32 GB.

## 8. Risks & gotchas

1. **gemma-4 is ~2 weeks old** — the llama.cpp SYCL build may not support the
   architecture yet. P4 gates this; fall back to E4B/Qwen2.5-Coder. (§3, §7)
2. **B580 is exclusive once passed through** — PCI passthrough means the host and
   other VMs lose it. Shared with the arr-stack Jellyfin plan (both consume the
   same passed-through card). (§2 P1)
3. **VRAM contention with Jellyfin** — 12 GB total; keep the model ≤8 GB so a
   simultaneous transcode (~0.5–1 GB) doesn't OOM. The 30B MoE options would
   violate this. (§3, §6)
4. **SYCL + Battlemage driver match** — the container's Intel compute runtime
   must match the `xe` driver/firmware in the VM; a mismatch silently falls back
   to CPU (very slow). Verify with `intel_gpu_top` showing activity during gen.
5. **Battlemage (Xe2) maturity** — same prerequisite as the arr-stack Jellyfin
   GPU check: kernel ≥ 6.12, `xe` driver, recent firmware/compute runtime.
6. **Single-user serving assumption** — the VM has ample vCPU (12) but the box is
   shared with Immich/arr/Forgejo; expect good single-user latency, not high
   concurrent throughput.
6. **On-demand loading is mandatory** — without it, the resident 8B model + Immich
   + the rest pressure the 28 GB VM. The pod must unload/scale-to-zero when idle.
7. **Model download size** — gemma-4-E4B-it GGUF is ~5 GB; store on the ZFS NFS
   share (`tank/llm-models`), snapshot before/after.

## 9. Open decisions

1. **Primary model** — confirm **gemma-4-E4B-it (8B)** (recommended for the 32 GB
   box + Immich), pending P4. The 12B is only viable if you later add RAM.
2. **Context window** — 8 K default; raise if you need long-repository context
   (costs VRAM for KV cache, eats into Jellyfin headroom).
3. **Editor integration** — Continue.dev / codecompanion.nvim / other? Determines
   whether FIM (`/infill`) or chat is the primary surface.
4. **Optional embeddings endpoint** — enable a small embedding model
   (e.g. nomic-embed) on the same pod for local RAG later?
