# Arr Stack Implementation Plan

> **Status: Design / pending decisions.** This is a design plan, not yet a
> task-by-task executable plan. Resolve the decisions in §13 and the
> prerequisites in §2 first, then convert this into a phased executable plan
> (see §12). No manifests are written until then.

**Goal:** Deploy a full \*arr media stack (Sonarr, Radarr, Prowlarr, Bazarr,
Seerr, Jellyfin) plus a VPN-protected qBittorrent download path, all
internal-only, on the `pk3s` master node.

**Architecture:** One shared `media` namespace on the master node. All arr apps
and Jellyfin mount a single shared NFS volume backed by the Proxmox host's ZFS
pool (so hardlinks between downloads and media work, and the media data never
moves off the storage host). qBittorrent runs as a sidecar in a Gluetun pod,
with all its egress forced through an AirVPN WireGuard tunnel by gluetun's
kill-switch firewall. Jellyfin transcodes on the Intel Arc B580 (Battlemage)
via VAAPI/QSV.

**Tech Stack:** raw Kubernetes manifests (no Helm charts — matches the `floci`
raw pattern), linuxserver.io images for arr apps, `jellyfin/jellyfin`,
`qmcgaw/gluetun`, `ghcr.io/seerr-team/seerr`, Traefik IngressRoute (CRD),
SealedSecrets for AirVPN credentials.

**Constraints (from design discussion):**
- Internal-only — `.local` hostnames, `web` entrypoint. No Cloudflare Tunnel.
- All pods pinned to the master node (`nodeSelector: k3s-master`); workers only
  have ~2 GiB RAM each.
- Secrets via SealedSecrets (AirVPN WireGuard creds). Plaintext never in git/chat.

**Cluster reality (corrected from capacity check):** `k3s-master` is a Proxmox
VM with **4 vCPU / ~8 GiB RAM** (workers are 1 vCPU / ~2 GiB each). The 8 TB
media library lives on the **Proxmox host's ZFS pool** — not on the master VM.
It is exposed to the master over NFS (no data migration required).

## 2. Prerequisites & blockers

| # | Item | Blocks | How |
|---|------|--------|-----|
| P1 | GPU verification on master | Jellyfin only | kernel ≥ 6.12, `xe` driver bound to B580, `vainfo` shows encode+decode |
| P2 | AirVPN server name/country | Gluetun manifest | user provides |
| P3 | AirVPN forwarded port | Gluetun + qBittorrent | user provides (AirVPN client area) |
| P4 | AirVPN WireGuard creds (private key, PSK, assigned IP) | Gluetun secret | **user seals locally with `kubeseal`** |
| P5 | `render` GID on master | Jellyfin supplementalGroups | `getent group render` on master |
| P6 | NFS export of ZFS dataset from Proxmox host | all arr apps + Jellyfin | see §4 — one-time host setup |

Non-GPU apps (storage, arr managers, VPN pod) can be built as soon as P2–P4
are in. Jellyfin waits for P1 + P5.

### GPU verification commands (run on the master)

```bash
lspci -nn | grep -iE 'vga|3d|display'                      # card detected
lspci -nnk | grep -iA3 'vga\|3d\|display'                  # driver bound (must be 'xe')
uname -r && lsmod | grep -iE '^xe|^i915'
ls -l /dev/dri/                                            # need card0 + renderD128
vainfo 2>&1 | grep -iE 'VA-API|encode|decode|profile'      # media engines
getent group render                                        # GID for Jellyfin
```

**Decision rule:** proceed with Jellyfin only if `vainfo` shows H.264/H.265/AV1
encode+decode profiles and `/dev/dri/renderD128` exists with the `xe` driver
bound. If not, the kernel/firmware is too old for Battlemage.

│ Pod: qbittorrent                                                │

│  └ qbittorrent     (shares netns → all egress via VPN)          │
│      Service: qbittorrent:8080 (WebUI, cluster-internal)        │
├─────────────────────────────────────────────────────────────────┤
│ Pod: sonarr       ─┐                                            │
│ Pod: radarr       ─┼─ all mount the SAME NFS PV (from Proxmox    │
│ Pod: prowlarr     ─┤  ZFS) — hardlinks between downloads & media │
│ Pod: bazarr       ─┤                                            │
│ Pod: seerr        ─┤                                            │
│ Pod: jellyfin     ─┘  (+ /dev/dri GPU, pending P1)              │

**Network flow:**
- Users → Traefik (`web` :30080) → each app's ClusterIP → app pod (LAN only)
- Sonarr/Radarr → `qbittorrent.media.svc:8080` → enters gluetun netns → qBittorrent
- qBittorrent egress → gluetun tun0 → AirVPN (kill-switch blocks all non-tunnel egress)

## 4. Storage design — NFS from the Proxmox ZFS pool (no data migration)

The 8 TB media library already lives on the Proxmox host's **ZFS pool** on the
same physical box as the cluster. It is **not copied** anywhere — the existing
ZFS dataset is NFS-exported to the master VM. This keeps compute (k3s) and
storage (ZFS) cleanly separated, gives free ZFS snapshots for rollback, and
requires zero data movement.

```
Proxmox host  (ZFS pool: tank/media — already holds the 8 TB library)
        │  NFS export over the VM bridge
        ▼
k3s-master VM  →  NFS PV mounted at /mnt/media  (shared by all arr pods + Jellyfin)
        │
   Fresh Jellyfin/arr apps import the existing library (no re-download)
```

### Why a single shared volume still matters (hardlinks)

qBittorrent, Sonarr/Radarr, and Jellyfin all mount the **same** NFS export.
When a torrent completes, Sonarr/Radarr "move" it into the media folder via a
**hardlink** (instant, zero extra disk). Hardlinks only work **within one
filesystem** — so `/downloads` and `/movies`/`/tv` must live in the same ZFS
dataset (one NFS export = one filesystem). NFSv4 supports hardlinks, so this
works over the wire. ✓

### Layout on the ZFS dataset

```
tank/media  (single dataset, NFS-exported, mounted at /mnt/media in the master)
  /mnt/media/movies      ← Radarr manages, Jellyfin reads
  /mnt/media/tv          ← Sonarr manages, Jellyfin reads
  /mnt/media/downloads   ← qBittorrent writes, arr hardlink out

Per-app config PVCs (small, local-path on the master, 1–3 GiB each):
  sonarr-config, radarr-config, prowlarr-config, bazarr-config,
  seerr-config, jellyfin-config, qbittorrent-config
```

All arr containers run as a **common PUID/PGID** (proposed `1000:1000`).

### Proxmox host setup (one-time)

```bash
# 1. Identify the dataset holding the media
zfs list            # e.g. tank/media

# 2. Snapshot before anything else (free, instant, your rollback)
zfs snapshot tank/media@pre-migration

# 3. Set ownership to the PUID/PGID the arr stack will run as (metadata-only, fast)
chown -R 1000:1000 /tank/media

# 4. NFS-export to the cluster service CIDR (all_squash → uid 1000 matches containers)
#    /etc/exports:
#    /tank/media  10.43.0.0/16(rw,sync,no_subtree_check,anonuid=1000,anongid=1000,all_squash)
exportfs -ra

# 5. Optional: cap the dataset so the arr stack can't fill the whole pool
zfs set refquota=6T tank/media

# 6. Stop the old media-server VM/software on the host (data on ZFS is untouched)
```

### k3s side

Deploy an **NFS PV** (or `nfs-subdir-external-provisioner` for dynamic PVCs)
pointing at `<proxmox-host>:/tank/media`. Mount it as the shared `/mnt/media`
volume consumed by every arr pod and Jellyfin. Per-app config stays on small
`local-path` PVCs on the master VM.

### ZFS bonuses

- **Instant snapshots** before any bulk arr reorganization (`zfs snapshot`) —
  far safer than rsync-and-pray.
- **Scrub** catches bitrot in the media library over the years.
- **`refquota`** prevents the arr stack from starving Proxmox for space.

### NFS vs virtio-fs (alternative)

NFS is the primary recommendation: first-class k8s PVC support, flexible (other
VMs/clients can mount too), trivial over the in-box virtual bridge. Proxmox 8.x
can also **virtio-fs** a host directory into the VM (lower overhead, but surfaces
as a hostPath-style mount rather than a network PV). Start with NFS; switch only
if a bottleneck is measured.

## 6. qBittorrent + Gluetun pod design (the hard part)

**Pod topology:** one Deployment, two containers. qBittorrent shares gluetun's
network namespace automatically (pod-level). Gluetun holds the tunnel + iptables;
qBittorrent's traffic is forced through `tun0` by gluetun's firewall.

**Gluetun container:**
- Capabilities: `NET_ADMIN`; device `/dev/net/tun`; `SYS_MODULE` only if the
  WireGuard module isn't built into the kernel (modern kernels have it).
- Env (non-secret):
  - `VPN_SERVICE_PROVIDER=airvpn`
  - `VPN_TYPE=wireguard`
  - `SERVER_COUNTRIES=<yours>`  (from P2)
  - `FIREWALL_VPN_INPUT_PORTS=<your forwarded port>`  (from P3, for peers)
  - `FIREWALL_INPUT_PORTS=8080`  (LAN/cluster → qBittorrent WebUI)
  - `DOT=on`
- Env (secret, from SealedSecret `airvpn-wireguard`):
  - `WIREGUARD_PRIVATE_KEY`
  - `WIREGUARD_PRESHARED_KEY`
  - `WIREGUARD_ADDRESSES`
- Healthcheck: gluetun's `/v1/publicip/ip` (HTTP 9999) confirms the tunnel is
  up. qBittorrent's readiness should depend on gluetun being healthy.

**qBittorrent container:**
- Listens on 8080; the "incoming connections" port = **same as the AirVPN
  forwarded port**, so peers reach it through the tunnel.
- Exposed via `Service qbittorrent:8080` for both Traefik (WebUI) and
  Sonarr/Radarr API access.

**Kill-switch guarantee:** gluetun's firewall (on by default) drops everything
except the tunnel + explicitly opened LAN/cluster ports. If the VPN drops,
qBittorrent has no route to the internet → zero leak. This is the whole point.

**Critical integration risk — arr→qBittorrent through the VPN firewall:**
Sonarr/Radarr traffic to `qbittorrent:8080` arrives at the pod from the k3s
**pod CIDR** (`10.42.0.0/16`) and **service CIDR** (`10.43.0.0/16`), not the
LAN subnet. gluetun's firewall must permit these, or the arr apps cannot reach
the WebUI. Default gluetun allows the local subnet; the cluster CIDRs likely
need explicit allowance. **Flag for verification at deploy time (Phase 1 gate).**

## 7. Jellyfin GPU design (B580, Battlemage)

- Mount `/dev/dri` (hostPath) into the Jellyfin container.
- `supplementalGroups: [<render GID from P5>]`.
- `JELLYFIN_HW_ACCEL=vaapi` (or `qsv`); enable in Playback → Transcoding.
- **Driver risk:** the B580 is Xe2-HPG and uses the **`xe`** kernel driver
  (not `i915`). The Jellyfin container bundles `intel-media-va-driver`; verify
  it's recent enough for Battlemage, else mount host driver/libraries. If
  `vainfo` inside the container doesn't list encode/decode profiles, transcode
  silently falls back to CPU (caught by P1).
- Transcode cache: small PVC or tmpfs at `/transcode`.

## 8. Secrets

Only one secret: `airvpn-wireguard` (in `media` ns), SealedSecret, containing
the three WireGuard fields. Sealed locally via the AGENTS.md `kubeseal` pattern
— plaintext never enters chat or git:

```bash
cd ~/Projects/homelab-apps
printf 'Private key: '; IFS= read -rs PRIVKEY; echo
printf 'apiVersion: v1\nkind: Secret\nmetadata:\n  name: airvpn-wireguard\n  namespace: media\ntype: Opaque\nstringData:\n  WIREGUARD_PRIVATE_KEY: %s\n  WIREGUARD_PRESHARED_KEY: <psk>\n  WIREGUARD_ADDRESSES: <addr>\n' "$PRIVKEY" \
  | kubeseal --controller-name sealed-secrets --controller-namespace sealed-secrets \
             --format yaml --namespace media \
  > clusters/pk3s/media/qbittorrent/sealedsecret-airvpn.yaml
unset PRIVKEY
```

Everything else (inter-app API keys, Jellyfin admin) is configured at runtime
through the WebUIs, not committed.

## 10. Resource budget (all on master)

| | Requests | Limits |
|---|---|---|
| **CPU** | ~1.55 cores | ~6 cores |
| **RAM** | ~3.3 GiB | ~8 GiB |

Master is a Proxmox VM with **4 vCPU / ~8 GiB**, currently using ~1.4 cores
requested / ~2.1 GiB (35% CPU, 26% RAM allocated). The arr stack's ~1.55 core /
~3.3 GiB requests fit in the remaining headroom, but it's **snug** — limits
(6 cores / 8 GiB) would oversubscribe. Keep limits modest and rely on the B580
for transcodes so Jellyfin doesn't spike the CPU. Storage I/O goes over NFS to
the ZFS pool, not through the master's local disk.

## 11. Risks & gotchas

1. **Hardlink filesystem** — `/downloads` and `/movies`/`/tv` must be in the
   same ZFS dataset (one NFS export) for hardlinks to work, or accept doubled
   disk usage. Satisfied by the §4 layout. (§4)
2. **NFS permissions / root-squash** — use `all_squash,anonuid=1000,anongid=1000`
   on the export so every container write maps to UID 1000 (PUID/PGID). A
   mismatch here is the classic "arr apps can't write to the media folder" bug.
3. **NFS over the VM bridge performance** — fine for a homelab, but Jellyfin
   direct-play streams many files over NFS; switch to virtio-fs only if a real
   bottleneck is measured (don't preemptively optimize).
4. **Cluster CIDR through gluetun firewall** — arr apps may not reach the
   qBittorrent WebUI unless `10.42.0.0/16` + `10.43.0.0/16` are permitted. (§6)
5. **Battlemage driver in container** — Jellyfin image may lack Xe2-capable
   `intel-media-va-driver`; transcode silently falls back to CPU. Verify with
   `vainfo` inside the pod. (§7, P1)
6. **Connectable port** — if the AirVPN forwarded port isn't set identically in
   gluetun's `FIREWALL_VPN_INPUT_PORTS` and qBittorrent's listen port,
   qBittorrent stays "not connectable" → poor seeding.
7. **Disk fill (ZFS, not master)** — qBittorrent seeding + unpacking can fill
   the `tank/media` dataset. Set a `refquota` and add an Alertmanager rule for
   ZFS pool >80% (Alertmanager is already deployed).
8. **FlareSolverr** — not selected. If any Prowlarr indexers are
   Cloudflare-protected (common for torrent trackers), add it later; without it
   those indexers return CF challenges and fail.
9. **PUID/PGID consistency** — every linuxserver container must use the same
   PUID/PGID (1000:1000), matching the NFS export's `anonuid`/`anongid` and the
   `chown` on the dataset. A mismatch breaks writes and hardlinks.
